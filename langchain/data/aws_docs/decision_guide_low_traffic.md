# 저트래픽 개인 프로젝트를 위한 컴퓨팅 서비스 선택 가이드

트래픽이 적은 개인 프로젝트(포트폴리오, 토이 프로젝트, 초기 단계 서비스 등)에서 EC2, Lambda, Fargate 중 무엇을 선택할지는 다음 기준으로 판단할 수 있다.

## 판단 기준

1. **호출 빈도**: 하루 수십~수백 건 이하로 간헐적이라면 Lambda의 요청 단위 과금이 가장 저렴하다. EC2처럼 상시 켜두는 방식은 실제 사용 시간 대비 낭비가 크다.
2. **실행 시간 제약**: 각 요청 처리가 15분을 넘지 않는다면 Lambda가 후보가 된다. 더 길게 실행되어야 하거나 WebSocket처럼 상시 연결이 필요하면 Fargate나 EC2를 고려해야 한다.
3. **기존 패키징 형태**: 이미 Docker 이미지로 패키징되어 있고 커스텀 런타임/장시간 프로세스가 필요하면 Fargate가 Lambda보다 적합할 수 있다.
4. **콜드 스타트 민감도**: 응답 지연에 민감한 서비스라면 Lambda의 콜드 스타트를 고려해 프로비저닝된 동시성(Provisioned Concurrency)을 쓰거나 Fargate/EC2를 검토한다.
5. **운영 부담**: 서버 패치, 스케일링, 이중화를 직접 하고 싶지 않다면 Lambda나 Fargate 같은 관리형 서비스가 유리하다.

## 결론 요약

- 트래픽이 적고 간헐적이며 요청이 짧게 끝난다 → **Lambda** 우선 고려.
- 컨테이너 기반이고 실행 시간이 길거나 커스텀 런타임이 필요하다 → **Fargate** 고려.
- 트래픽이 예측 가능하고 꾸준하거나, 세밀한 인프라 제어가 필요하다 → **EC2** 고려.

이 판단은 절대적 규칙이 아니라 트래픽 패턴, 비용 민감도, 운영 역량에 따라 달라질 수 있으므로, Well-Architected Framework의 비용 최적화 원칙(수요에 맞게 지출하기)을 함께 참고해야 한다.
