# AWS Well-Architected Framework: 비용 최적화 축약 가이드

AWS Well-Architected Framework의 비용 최적화 기둥(Cost Optimization Pillar)은 불필요한 비용을 피하면서 비즈니스 목표를 달성하는 것을 목표로 한다. 핵심 원칙은 다음과 같다.

## 핵심 원칙

1. **수요에 맞게 지출하기(Match supply and demand)**: 트래픽이 변동하는 워크로드는 필요한 만큼만 리소스를 사용하도록 자동 확장(Auto Scaling)하거나, 아예 요청 단위로 과금되는 서버리스(Lambda)를 사용한다.
2. **올바른 요금 모델 선택**: 사용량이 예측 가능하고 꾸준하면 Reserved/Savings Plans, 변동성이 크고 중단 허용 가능한 작업은 Spot, 트래픽이 적거나 간헐적이면 On-Demand 서버리스가 유리하다.
3. **관리형 서비스 활용**: 인프라 운영(패치, 스케일링, 이중화)을 직접 하지 않아도 되는 관리형 서비스(Lambda, Fargate, RDS 등)를 사용하면 운영 비용(엔지니어링 시간)도 절감된다.
4. **전체 비용을 측정하고 인식하기**: 컴퓨팅 비용뿐 아니라 데이터 전송, 스토리지, 유휴 리소스 비용까지 포함해 총소유비용(TCO)을 판단해야 한다.

## 컴퓨팅 서비스 선택에 적용하기

트래픽 패턴이 비용 최적화의 가장 중요한 판단 기준이다.

- 트래픽이 거의 없거나 매우 간헐적 → 서버리스(Lambda)가 유휴 비용이 없어 가장 경제적인 경우가 많다.
- 트래픽이 꾸준하고 예측 가능 → EC2 + Reserved/Savings Plans가 장기적으로 저렴할 수 있다.
- 컨테이너 기반이며 운영 부담을 줄이고 싶고, 트래픽이 중간 정도 → Fargate가 EC2 직접 관리보다 유리할 수 있다.

## 흔한 비용 낭비 패턴

- 사용하지 않는데 계속 켜져 있는 EC2 인스턴스
- 트래픽이 거의 없는 서비스에 과도하게 큰 인스턴스 타입을 선택하는 경우(오버 프로비저닝)
- Free Tier 한도를 인지하지 못하고 유사한 유료 리소스를 중복 사용하는 경우
